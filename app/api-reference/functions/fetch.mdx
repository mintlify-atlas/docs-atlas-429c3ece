Next.js extends the [Web `fetch()` API](https://developer.mozilla.org/docs/Web/API/Fetch_API) to allow each server-side request to set its own persistent caching and revalidation semantics.

You can call `fetch` with `async/await` directly within Server Components.

```tsx filename="app/page.tsx"
export default async function Page() {
  let data = await fetch('https://api.vercel.app/blog')
  let posts = await data.json()
  return (
    <ul>
      {posts.map((post) => (
        <li key={post.id}>{post.title}</li>
      ))}
    </ul>
  )
}
```

## `fetch(url, options)`

Next.js extends the native `fetch` API. You can use any [native fetch options](https://developer.mozilla.org/docs/Web/API/fetch#parameters) in addition to the Next.js-specific options below.

### `options.cache`

Configure how the request interacts with Next.js caching.

```ts
fetch('https://...', { cache: 'force-cache' | 'no-store' })
```

- **`auto no cache`** (default): Fetches fresh on every request in development; fetches once during `next build` for statically prerendered routes. If Request-time APIs are detected, fetches on every request.
- **`no-store`**: Always fetches from the remote server, even for statically prerendered routes.
- **`force-cache`**: Looks for a matching request in the server-side cache. Returns cached data if fresh; otherwise fetches and updates the cache.

### `options.next.revalidate`

```ts
fetch('https://...', { next: { revalidate: false | 0 | number } })
```

Sets the cache lifetime of a resource in seconds.

- `false`: Cache the resource indefinitely. Equivalent to `revalidate: Infinity`.
- `0`: Prevent the resource from being cached.
- `number`: Specify a cache lifetime of at most `n` seconds.

<Note>If an individual `fetch()` request sets a `revalidate` lower than the route's default, the whole route revalidation interval decreases. If two fetches to the same URL have different `revalidate` values, the lower one is used. Conflicting options like `{ revalidate: 3600, cache: 'no-store' }` are not allowed.</Note>

### `options.next.tags`

```ts
fetch('https://...', { next: { tags: ['collection'] } })
```

Set cache tags on a resource for on-demand revalidation with [`revalidateTag`](/app/api-reference/functions/revalidate-tag). Max tag length: 256 characters. Max tags: 128.

## Memoization

`fetch` requests using `GET` with the same URL and options are automatically memoized during a server render pass. If the same `fetch` is called in multiple Server Components, layouts, or pages, Next.js executes it only once and shares the result.

To opt out, pass an `AbortController` signal:

```js
const { signal } = new AbortController()
fetch(url, { signal })
```

<Note>Memoization does not apply in Route Handlers, as they are not part of the React component tree.</Note>

## Troubleshooting

### Fresh data not showing in development

Next.js caches `fetch` responses across Hot Module Replacement (HMR) in local development for faster responses. The [HMR cache](/app/api-reference/config/next-config-js#servercomponentshmrcache) applies to all fetch requests including those with `no-store`. The cache is cleared on navigation or full-page reloads.

See `serverComponentsHmrCache` docs for more information.
