Next.js comes with built-in TypeScript, automatically installing necessary packages and configuring settings when you create a new project with `create-next-app`.

To add TypeScript to an existing project, rename a file to `.ts` or `.tsx`, then run `next dev` or `next build`. Next.js installs required dependencies and adds a `tsconfig.json` with recommended settings.

## IDE plugin

Next.js includes a custom TypeScript plugin and type checker for VSCode and compatible editors.

Enable in VS Code:

<Steps>
  <Step title="Open command palette">
    Press `Ctrl/⌘ + Shift + P`.
  </Step>
  <Step title="Select TypeScript version">
    Search for "TypeScript: Select TypeScript Version" and select "Use Workspace Version".
  </Step>
</Steps>

The plugin helps with:

- Warning on invalid values for segment config options.
- Showing available options and in-context documentation.
- Ensuring `'use client'` is used correctly.
- Ensuring client hooks like `useState` are only used in Client Components.

## End-to-end type safety

The App Router has enhanced type safety:

1. **No serialization between fetching and page**: `fetch` directly in components returns typed data without serialization.
2. **Streamlined data flow**: With root layouts instead of `_app`, data flow between components is easier to type.

```tsx filename="app/page.tsx"
async function getData() {
  const res = await fetch('https://api.example.com/...')
  // The return value is *not* serialized
  // You can return Date, Map, Set, etc.
  return res.json()
}

export default async function Page() {
  const name = await getData()
  return '...'
}
```

## Route-aware type helpers

Next.js generates global helpers for App Router route types. Available without imports, generated during `next dev`, `next build`, or `next typegen`:

- [`PageProps`](/app/api-reference/file-conventions/page#page-props-helper) — strongly typed `params` and `searchParams` for pages.
- [`LayoutProps`](/app/api-reference/file-conventions/layout#layout-props-helper) — strongly typed `params` and named slots for layouts.
- [`RouteContext`](/app/api-reference/file-conventions/route#route-context-helper) — strongly typed `params` for route handlers.

## `next-env.d.ts`

Next.js generates a `next-env.d.ts` file in your project root. This file references Next.js type definitions, allowing TypeScript to recognize non-code imports (images, stylesheets) and Next.js-specific types.

Running `next dev`, `next build`, or `next typegen` regenerates this file.

<Note>Add `next-env.d.ts` to your `.gitignore`. The file must be in the `include` array of your `tsconfig.json` (done automatically by `create-next-app`).</Note>

## Type checking `next.config.ts`

```ts filename="next.config.ts"
import type { NextConfig } from 'next'

const nextConfig: NextConfig = {
  /* config options here */
}

export default nextConfig
```

## Incremental type checking

Next.js supports incremental type checking when `incremental: true` is in `tsconfig.json`. This improves type checking performance in large applications.

## Ignoring TypeScript errors

<Warning>Enabling this setting skips TypeScript error checking at build time. Failing type checks are a security risk.</Warning>

If you need to dangerously ship production code with TypeScript errors:

```js filename="next.config.js"
module.exports = {
  typescript: {
    ignoreBuildErrors: true,
  },
}
```

## Custom TypeScript config path

```js filename="next.config.js"
module.exports = {
  typescript: {
    tsconfigPath: 'tsconfig.build.json',
  },
}
```
