## App Router

The **App Router** (`app/` directory) is the recommended way to build Next.js applications. It is built on top of React Server Components and introduces a new file-system routing convention.

### File-system conventions

Every folder under `app/` maps to a URL segment. Special files control how each segment is rendered:

| File | Purpose |
|---|---|
| `page.tsx` | The unique UI for a route segment, makes it publicly accessible |
| `layout.tsx` | Shared UI that wraps child segments; preserves state across navigations |
| `loading.tsx` | Instant loading UI using React `Suspense` |
| `error.tsx` | Error boundary UI for a segment and its children |
| `not-found.tsx` | UI rendered when `notFound()` is thrown |
| `route.ts` | Server-side API endpoint (replaces `pages/api/`) |
| `proxy.ts` | Runs before a request is matched to a route (formerly `middleware.ts` in Next.js < 16) |

### Nested layouts

Layouts can be nested. A root layout (`app/layout.tsx`) wraps the entire app, and each route segment can define its own layout that wraps its children. Layouts are React Server Components by default and do **not** re-render on navigation — only the part of the tree that changes is re-rendered.

```tsx filename="app/dashboard/layout.tsx"
export default function DashboardLayout({
  children,
}: {
  children: React.ReactNode
}) {
  return (
    <section>
      <nav>{/* sidebar nav */}</nav>
      <main>{children}</main>
    </section>
  )
}
```

### React Server Components

All components inside `app/` are **React Server Components (RSC)** by default. They render on the server, and their output — a serialized component tree called the **React Server Component Payload (RSC Payload)** — is sent to the client. This means:

- Server components have direct access to databases, filesystems, and server-only secrets.
- They do not add JavaScript to the client bundle.
- They can `async/await` directly without any additional libraries.

To use browser APIs, event handlers, or React state and effects, add `"use client"` at the top of the file. This marks the component as a **Client Component** and creates a boundary between server and client code.

```tsx filename="app/counter.tsx"
'use client'

import { useState } from 'react'

export default function Counter() {
  const [count, setCount] = useState(0)
  return <button onClick={() => setCount(count + 1)}>{count}</button>
}
```

## Pages Router

The **Pages Router** (`pages/` directory) is the original Next.js routing model, still fully supported. Each file in `pages/` automatically becomes a route.

| Export | Rendering strategy |
|---|---|
| `getStaticProps` | Static Site Generation (SSG) — rendered at build time |
| `getStaticPaths` + `getStaticProps` | SSG with dynamic routes — generates static pages for each path |
| `getServerSideProps` | Server-Side Rendering (SSR) — rendered on every request |
| (none) | Static export — rendered at build time with no data fetching |

<Note>
The App Router and Pages Router can coexist in the same project. Files under `app/` take precedence over `pages/` when routes conflict.
</Note>

## Build Pipeline

Next.js uses a two-phase build system: **compilation** (per-file transforms via SWC) and **bundling** (module graph via Turbopack or webpack).

### SWC Compiler

Next.js ships a Rust-based compiler built on [SWC](https://swc.rs/) that handles:

- **JSX and TypeScript transforms** — converting JSX and TypeScript to plain JavaScript
- **Dead code elimination** — stripping `"use client"` / `"use server"` boundaries
- **Minification** — replacing Terser (17x faster)
- **Custom transforms** — styled-components, Relay, Emotion, and more

The compiler lives in `crates/next-custom-transforms/` and is exposed to Node.js via native bindings in `packages/next-swc/`. When a `.babelrc` is detected, Next.js falls back to Babel automatically.

### Turbopack (default bundler)

Turbopack is the default bundler for both `next dev` and `next build`. It is written in Rust and lives in the `turbopack/` directory of the monorepo.

Key characteristics:

- **Incremental by default** — only re-bundles the modules that changed
- **Parallel compilation** — leverages all CPU cores
- **Demand-driven** — only compiles code actually requested by the browser in dev

```bash
# Force webpack instead of Turbopack
next build --webpack
next dev --webpack
```

### Build entry points

```
packages/next/src/cli/next-build.ts   →   packages/next/src/build/index.ts
packages/next/src/cli/next-dev.ts     →   packages/next/src/server/dev/next-dev-server.ts
packages/next/src/cli/next-start.ts   →   packages/next/src/server/next-server.ts
```

## Server vs Client Rendering

Next.js supports multiple rendering strategies, which can be mixed at the component level in the App Router.

<AccordionGroup>
  <Accordion title="Static Rendering">
    Components (and pages) are rendered at **build time**. The output is cached and can be served from a CDN. This is the default for App Router routes that do not use dynamic data.

    ```tsx
    // This page will be statically rendered at build time
    export default async function Page() {
      const data = await fetch('https://api.example.com/data', {
        cache: 'force-cache',
      })
      return <main>{/* ... */}</main>
    }
    ```
  </Accordion>
  <Accordion title="Dynamic Rendering">
    A route is rendered **on each request** when dynamic functions are used (`cookies()`, `headers()`, `searchParams`) or when `fetch` uses `{ cache: 'no-store' }`.

    ```tsx
    import { cookies } from 'next/headers'

    export default async function Page() {
      const cookieStore = await cookies()
      const theme = cookieStore.get('theme')
      return <main>{/* ... */}</main>
    }
    ```
  </Accordion>
  <Accordion title="Streaming">
    Streaming allows the server to progressively send rendered HTML to the client as components resolve. Wrap slower parts of the UI in `<Suspense>` to stream their content independently.

    ```tsx
    import { Suspense } from 'react'
    import { SlowComponent } from './slow-component'

    export default function Page() {
      return (
        <main>
          <h1>Dashboard</h1>
          <Suspense fallback={<p>Loading...</p>}>
            <SlowComponent />
          </Suspense>
        </main>
      )
    }
    ```
  </Accordion>
  <Accordion title="Server Actions">
    Server Actions are async functions marked with `'use server'` that run on the server. They can be called directly from Client Components to handle form submissions and data mutations.

    ```tsx
    async function createUser(formData: FormData) {
      'use server'
      const name = formData.get('name')
      await db.users.create({ name })
    }
    ```
  </Accordion>
</AccordionGroup>

## Edge Runtime

Next.js supports three runtime environments:

| Runtime | Description | Use case |
|---|---|---|
| **Node.js** (default) | Full Node.js APIs | Server components, API routes, SSR |
| **Edge** | Web APIs only, no Node.js builtins | Middleware, lightweight API routes |
| **Browser** | Standard browser APIs | Client components |

### Edge runtime characteristics

The edge runtime (`packages/next/src/server/web/`) is a subset of standard Web APIs:

- Runs via the V8 engine (no Node.js)
- Faster cold starts than Node.js functions
- Globally distributed — runs close to the user
- Supports: `fetch`, `Request`, `Response`, `URL`, `TextEncoder`, `crypto`, `ReadableStream`, and more
- Does **not** support: `fs`, `path`, native Node.js modules, or most npm packages that depend on Node.js internals

### Opting into the edge runtime

```tsx filename="app/api/hello/route.ts"
export const runtime = 'edge'

export function GET() {
  return new Response('Hello from the edge!')
}
```

```tsx filename="proxy.ts"
import { NextResponse } from 'next/server'
import type { NextRequest } from 'next/server'

export function proxy(request: NextRequest) {
  return NextResponse.redirect(new URL('/home', request.url))
}

export const config = {
  matcher: '/about/:path*',
}
```

<Warning>
Proxy always runs in the edge runtime. You cannot use Node.js APIs in `proxy.ts`.
</Warning>

## Request Lifecycle

<Steps>
  <Step title="Request arrives">
    A request hits the Next.js server (Node.js process or edge worker). The router server (`packages/next/src/server/lib/router-server.ts`) matches the URL to a route.
  </Step>
  <Step title="Proxy runs">
    If a `proxy.ts` file matches the request path, it executes in the edge runtime before any route handler. Proxy can redirect, rewrite, or modify request/response headers.
  </Step>
  <Step title="Route is rendered">
    For App Router routes: the server renders the component tree, executing Server Components and collecting their RSC Payload. For Pages Router routes: `getServerSideProps` or `getStaticProps` runs first, then the page component is rendered.
  </Step>
  <Step title="Streaming response">
    HTML is progressively streamed to the client. The browser can start displaying content before the full response is received. Client Components are hydrated as their JavaScript bundles arrive.
  </Step>
  <Step title="Client-side navigation">
    After the initial load, `next/link` prefetches subsequent pages. Navigations happen client-side using the RSC Payload — no full page reload occurs.
  </Step>
</Steps>
