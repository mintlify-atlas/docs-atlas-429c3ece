Next.js offers multiple rendering strategies per page. You can choose the most appropriate approach for your data requirements.

## Server-side rendering (SSR)

Also referred to as "Dynamic Rendering". When a page uses Server-side Rendering, the HTML is generated on **each request**.

Export an `async` function called `getServerSideProps` from a page to enable SSR:

```jsx
export default function Page({ data }) {
  // Render data...
}

// This gets called on every request
export async function getServerSideProps() {
  // Fetch data from external API
  const res = await fetch(`https://.../data`)
  const data = await res.json()

  // Pass data to the page via props
  return { props: { data } }
}
```

Use SSR when your page relies on frequently updated data, or data that can only be known at request time (such as authentication headers or geolocation).

## Static site generation (SSG)

When a page uses Static Generation, the HTML is generated at **build time**. The HTML is then reused on each request and can be cached by a CDN.

### Static generation without data

By default, Next.js pre-renders pages using Static Generation without fetching data:

```jsx
function About() {
  return <div>About</div>
}

export default About
```

### Static generation with data

**Content depends on external data** — use `getStaticProps`:

```jsx
export default function Blog({ posts }) {
  return (
    <ul>
      {posts.map((post) => (
        <li>{post.title}</li>
      ))}
    </ul>
  )
}

// This function gets called at build time
export async function getStaticProps() {
  const res = await fetch('https://.../posts')
  const posts = await res.json()

  return {
    props: {
      posts,
    },
  }
}
```

**Paths depend on external data** — use `getStaticPaths` with `getStaticProps`:

```jsx
// This function gets called at build time
export async function getStaticPaths() {
  const res = await fetch('https://.../posts')
  const posts = await res.json()

  const paths = posts.map((post) => ({
    params: { id: post.id },
  }))

  return { paths, fallback: false }
}

export async function getStaticProps({ params }) {
  const res = await fetch(`https://.../posts/${params.id}`)
  const post = await res.json()

  return { props: { post } }
}

export default function Post({ post }) {
  // Render post...
}
```

### When to use static generation

Use Static Generation whenever possible because your page can be built once and served by CDN, which is much faster than server rendering on every request.

Good candidates for SSG include:

- Marketing pages
- Blog posts and portfolios
- E-commerce product listings
- Help and documentation

<Tip>
  Ask yourself: "Can I pre-render this page **ahead** of a user's request?" If yes, choose Static Generation.
</Tip>

## Automatic static optimization

Next.js automatically determines that a page is static if it has no blocking data requirements (no `getServerSideProps` or `getInitialProps`). Optimized pages can be cached and served from multiple CDN locations.

When a page is statically optimized, its query parameters are not available during the initial render. During hydration, Next.js will update the application to provide the `query` object.

## Client-side rendering (CSR)

In Client-Side Rendering, the browser downloads a minimal HTML page and JavaScript. The JavaScript is then used to update the DOM and render the page.

In Next.js, there are two ways to implement CSR:

1. Use `useEffect()` inside your pages instead of server-side rendering methods
2. Use a data fetching library like [SWR](https://swr.vercel.app/) or [TanStack Query](https://tanstack.com/query/latest/) (recommended)

### Using `useEffect`

```jsx filename="pages/index.js"
import React, { useState, useEffect } from 'react'

export function Page() {
  const [data, setData] = useState(null)

  useEffect(() => {
    const fetchData = async () => {
      const response = await fetch('https://api.example.com/data')
      if (!response.ok) {
        throw new Error(`HTTP error! status: ${response.status}`)
      }
      const result = await response.json()
      setData(result)
    }

    fetchData().catch((e) => {
      console.error('An error occurred while fetching the data: ', e)
    })
  }, [])

  return <p>{data ? `Your data: ${data}` : 'Loading...'}</p>
}
```

### Using SWR

```jsx filename="pages/index.js"
import useSWR from 'swr'

export function Page() {
  const { data, error, isLoading } = useSWR(
    'https://api.example.com/data',
    fetcher
  )

  if (error) return <p>Failed to load.</p>
  if (isLoading) return <p>Loading...</p>

  return <p>Your Data: {data}</p>
}
```

<Note>
  CSR can impact SEO since some crawlers may not execute JavaScript. Next.js promotes a hybrid approach that lets you combine SSR, SSG, and CSR depending on the needs of each page.
</Note>

## Comparison

| Strategy | When HTML is generated | Data freshness | Performance |
|----------|----------------------|----------------|-------------|
| SSG | Build time | Stale until rebuild | Fastest (CDN) |
| SSR | Each request | Always fresh | Slower (server) |
| CSR | Client-side | Always fresh | Depends on client |
| ISR | Build time + revalidation | Controlled freshness | Fast (CDN + revalidation) |
